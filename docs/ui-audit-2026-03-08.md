## Redundancy Map

### Dark Paper style ‚Äî ~12 instances
Inline `style={{ backgroundColor: 'var(--mantine-color-dark-7)', border: '1px solid var(--mantine-color-dark-5)' }}`

Files: AbilityBlock (x1 per ability, in map), CombatSidebar/StatBox (x1), SpellcastingSection (x2), SensesSection (x1), Home (x4 cards), LoadCharacter (x1), CharacterSheet header Paper (x1)

**Fix:** Extract to `src/theme/styles.ts` as `darkPaperStyle`, or better: add Paper default in theme `components` overrides.

### NumberInput type guard ‚Äî ~20 instances
`typeof v === 'number' ? v : <fallback>`

Files: CombatSidebar (x10), HitDiceSection (x3), InventorySection (x3), SpellcastingSection (x2), CurrencySection (x5), ConditionsSection (x1)

**Fix:** `numOrDefault(v, fallback)` in `src/utils/`.

### Centered input styles ‚Äî ~19 instances
`styles={{ input: { textAlign: 'center' } }}` and variants with `fontWeight`/`fontSize`/`padding`

Files: CombatSidebar (x8, 4 with fontSize:18), HitDiceSection (x3), InventorySection (x3), SpellcastingSection (x4, with padding:2), CurrencySection (x5)

**Fix:** Constants in `src/theme/styles.ts`: `centeredInputStyles`, `centeredBoldInputStyles`, `centeredCompactInputStyles`.

### CRUD list operations ‚Äî 3 files with identical pattern
```ts
function updateX(index, partial) { onChange(list.map((item, i) => i === index ? {...item, ...partial} : item)); }
function addX() { onChange([...list, DEFAULT]); }
function removeX(index) { onChange(list.filter((_, i) => i !== index)); }
```

Files: HitDiceSection, AttacksSection, ResourcesSection

**Fix:** `useListCrud<T>(list, onChange, defaultItem)` hook returning `{ update, add, remove }`.

### Array toggle ‚Äî ~6 instances
`arr.includes(item) ? arr.filter(x => x !== item) : [...arr, item]`

Files: AbilityBlock/toggleSaveProf, SkillsSection/toggleProf, SkillsSection/toggleExpertise, ConditionsSection, ProficienciesSection

**Fix:** `toggleArrayItem(arr, item)` in `src/utils/`.

### Remove button ‚Äî 2 variants, 7 files
- ActionIcon variant: `<ActionIcon size="xs" variant="subtle" color="red">&times;</ActionIcon>` ‚Äî HitDice, Attacks, Resources, Spells, Inventory
- Badge variant: `<Badge variant="light" color="red" size="xs" style={{ cursor: 'pointer' }}>&times;</Badge>` ‚Äî Conditions, Proficiencies

**Fix:** Single `RemoveButton` component.

### Accordion styling ‚Äî 5 instances in FeaturesSection
`styles={{ item: { backgroundColor: 'var(--mantine-color-dark-7)' } }}` with `variant="separated" chevronPosition="left"`

**Fix:** Extract constant, or add Accordion default in theme `components`.

## Theme Drift Details

### AppShell.tsx ‚Äî raw hex colors
```ts
styles={{
  header: { backgroundColor: '#1e1a15', borderBottom: '1px solid #3d3227' },
  main: { backgroundColor: '#1a1612' },
}}
```
Should use: `var(--mantine-color-dark-9)` or `theme.other.parchmentBg` / `theme.other.parchmentBorder`.

### global.css ‚Äî duplicated hex
| Hex | Theme token | CSS usage |
|-----|------------|-----------|
| `#1a1612` | `theme.other.parchmentBg` | `body` bg, scrollbar track |
| `#e3d5be` | `parchment[2]` | `body` text color |
| `#3d3227` | `theme.other.parchmentBorder` | scrollbar thumb |
| `#4d4237` | (no token) | scrollbar thumb hover |

Should use: `var(--mantine-color-parchment-2)` etc.

### CurrencySection.tsx ‚Äî 5 hardcoded coin colors
```ts
{ key: 'pp', label: 'PP', color: '#c0c0c0' },
{ key: 'gp', label: 'GP', color: '#d4a017' },
{ key: 'ep', label: 'EP', color: '#8a8a8a' },
{ key: 'sp', label: 'SP', color: '#a0a0a0' },
{ key: 'cp', label: 'CP', color: '#b87333' },
```

Should use: theme tokens or CSS variables.

### theme.other tokens ‚Äî defined but unused
- `parchmentBg: '#1a1612'` ‚Äî components use `var(--mantine-color-dark-7)` or raw hex instead
- `parchmentBgLight: '#2a2219'` ‚Äî never referenced anywhere
- `parchmentBorder: '#3d3227'` ‚Äî never referenced anywhere
- `textMuted: '#8a7e6e'` ‚Äî never referenced, components use `c="dimmed"` instead

### Missing theme config
- No `primaryShade` ‚Äî dark mode defaults to shade 8, may be too dark
- No `components` overrides ‚Äî Paper, Input, Table, Accordion all use Mantine's default blue-gray dark palette

## Complexity Hotspots

### CharacterSheet.tsx ‚Äî 854 lines
Contains:
- Full 3-column CSS Grid layout with hardcoded widths (`260px 1fr 240px`)
- Embedded `<style>` tag with media queries at `1023px` and `767px`
- `SectionTitle` sub-component (inline, not exported)
- `HEADER_HEIGHT = 110` constant (coupled to AppShell's `height: 56`)
- All character state management (load, save, auto-save, dirty tracking)
- Direct prop drilling to 15+ child components

**Decomposition target:** SheetHeader, SheetLayout, useCharacterSheet hook.

### FeaturesSection.tsx ‚Äî ~553 lines
Contains:
- 5 module-level utility functions: `isSubclassPlaceholder`, `collectOptionFeatureNames`, `findListChoiceBlocks`, `stripListChoiceEntries`, `extractEntryText`
- 5 Accordion blocks (race, background, class per level, feats, custom feats)
- Inline feature choice rendering with Radio groups
- `backgroundName` prop declared but **never used** (dead code)

**Decomposition target:** Move utils to `src/utils/features.ts`, split into RaceFeatures, ClassFeatures, FeatsList sub-components.

## UX Issues

1. **Right sidebar clipped** ‚Äî "DEATH SAVES" heading cut off in screenshots. The `260px 1fr 240px` grid may overflow viewport. Fix: use `minmax()` or reduce right column.

2. **HitDice row dense** ‚Äî 8 elements in one Group: Badge, NumberInput, 3x Text, 2x NumberInput, ActionIcon. Hard to scan. Fix: restructure into 2-row or grid layout.

3. **CombatSidebar wrapping** ‚Äî `flex: '1 1 100px'` with `wrap` produces unpredictable layouts at different widths. Fix: use Grid with explicit column counts at breakpoints.

4. **Right sidebar sections run together** ‚Äî no visual separators between Death Saves, Hit Dice, Conditions, Senses, Proficiencies. Fix: use SectionTitle or dividers.

5. **Spellslot table always 9 columns** ‚Äî even when most are unused (low-level characters). Fix: only show columns up to highest available slot level.

## Callback Signature Inconsistencies

| Pattern | Components |
|---------|-----------|
| `onChange(value: string)` | NotesSection |
| `onChange(field: string, value: string)` | PersonalitySection, ProficienciesSection |
| `onChange(partial: Partial<T>)` | CombatSidebar |
| `onChange(fullObject: T)` | DeathSavesSection, AppearanceSection |
| `onChange(list: T[])` | HitDiceSection, AttacksSection, ResourcesSection, SpellsSection, ConditionsSection, InventorySection |
| Two named callbacks | BackstorySection (`onBackstoryChange`, `onAlliesChange`) |
| Mixed (object + separate callbacks) | AbilityBlock (`onAbilitiesChange` + `onSavesChange`) |

**Not necessarily wrong** ‚Äî list vs scalar vs multi-field have different ergonomics ‚Äî but the inconsistency between PersonalitySection (generic field name) and BackstorySection (named callbacks) for structurally similar components is worth normalizing.
